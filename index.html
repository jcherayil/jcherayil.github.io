PixelDust

Supporting dynamic area of interest tagging in programming studies with eye tracking

Eye tracking studies are valuable for evaluating programming environments, but annotating what the programmer is looking at in a dynamic environment can be repetitive, timeconsuming, and error prone. Through a participatory design exercise with two eye tracking researchers, I identified three significant challenges: search, extraction of code, and annotating transient objects.

By applying computer vision algorithms to video traces, I developed a mixed-initiative system called PixelDust, which allows the researcher to train a system to recognize different objects on a screen. My preliminary results demonstrate the versatility of the approach; for example, the system can recognize return statements, method signatures, tool tips, and dialog boxes. This research won first place in the Student Research Competition of the 2016 SPLASH Conference in Amsterdam.

Biometric Authentication

Discovering the optimal login flow for Deloitte employees using biometrics

Until this project, our apps did not share a common strategy for multi-factor authentication. Striving for consistency in our suite of apps, I set out to find the optimal login experience for users in terms of security, design, and user experience, with the goal of applying that login flow to all apps moving forward.

First, I researched existing APIs that enable biometric authentication in order to understand the capabilities and limitations of the available libraries. With this in mind, I enumerated several possible login scenarios, each with a different flow to guide the user into the app, and created a demo app to showcase each scenario in action.

After presenting these demo apps to end users and receiving their feedback, I was able to narrow down my options and recommend a seamless login experience that was visually appealing, safe, and user-friendly. This login experience is now standard for all of our mobile apps moving forward.

Pre-Populated Expense Reports from Receipts

Leveraging OCR technology to turn receipts into itemized expense reports.

Filling out expense reports in Deloitte's existing expensing app is particularly time-consuming for hotel receipts, in which every charge (e.g. room charges, food expenses, taxes) must be itemized per day. Several employees report spending hours expensing hotel receipts even after just a few days of travel.

In this demo, we demonstrated the power of using optical character recognition (OCR) to extract all charges from a PDF receipt. Once extracted, our program sorts the charges by date and fills out the corresponding fields of an itemized expense report. The user need only validate the report and hit submit, turning hours of expensing into sheer seconds. This prototype app proved the success of OCR in facilitating and quickening the process of itemizing expenses from hotel stays.

NGCIO OneDrive Automation

Automating OneDrive account file methods

Each year, the Next Generation CIO Academy takes place at Deloitte University, training 100 soon-to-be CIOs for their roles. Participants are given countless reference materials over the course of the one-week academy via OneDrive accounts. In preparation for the event, academy organizers expect to log into each account, upload files, and delete files. After the event, academy organizers log into each account again, move all the files to a single folder, and share that folder with each participant.

Manually logging into 100 identical OneDrive accounts, sometimes more than once per day, is tedious and invites human error. Instead, we decided to automate the process by creating a desktop application that uploads, deletes, moves, and shares files. Given the broken skeleton of an existing tool that uploaded documents to multiple OneDrive accounts simultaneously, I rebuilt the existing functionalities of uploading files and deleting files to ensure they worked as expected. In addition to this, I added a new feature that moves all academy materials to one folder and automatically provides the academy organizers with an invitation link to send to the participants. This desktop tool significantly reduced the time it took for academy organizers to distribute reference materials.

Jira to ADO Work Item Migration

Automating work item migration from Jira to Azure DevOps

A work items provide the foundation for monitoring progress on the development of an app. Our team initially used Jira, an issue-tracking system, but ultimately decided to move to the more holistic Azure DevOps. However, no tool existed to migrate issues automatically between the two systems, so users had started to manually create new issues in ADO, pull up the old Jira issues side by side, and re-populate all the fields in ADO themselves.

To improve on this approach, I built a tool to automate the migration process. For each Jira issue, the tool fetched all the fields (e.g. type, status, priority, description, etc.), created a new issue in Azure DevOps, and then copied the old fields into the new issue. Apart from logging into both Jira and ADO, this console application takes care of the entire migration process, with changes reflected immediately in ADO within minutes.

Maintenance Mode

Putting apps in maintenance mode for a select group of users

Deloitte's app catalog contains a list of all apps along with basic information about the app. If developers want to make a change to an app, they visit the app catalog and put the app under maintenance mode, which blocks all users from entering the app and displays a pop-up about how long the maintenance will last. However, there are times when changes need only be made for certain groups of users - for example, a change in an app might be specific to US users only, not all users worldwide. In this case, we want to be able to put an app under maintenance mode solely for those specific users.

The result of this project was an app catalog that made it possible to filter through groups of users when putting an app in maintenance mode. This required front-end and back-end ASP.NET work, testing the feature with various API calls, and finally staging the site in several different environments to see how it performed.
